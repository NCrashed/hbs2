# Quickstart

This walkthrough takes you from a fresh install to a git repository
that has been pushed into hbs2 and cloned back out. It runs entirely
on a single machine in two terminals and needs no external peers.

If you have not installed hbs2, see [`INSTALL.md`](INSTALL.md) first.

If you came from the legacy `hbs2` monolithic CLI, the old commands
are gone. The `hbs2-cli` and `hbs2-git3` tools replace them. See
[`docs/CLI_MIGRATION.md`](docs/CLI_MIGRATION.md) for the full mapping.

## 1. Generate a peer config

Each `hbs2-peer` instance reads its config from a directory. The
default location is `~/.config/hbs2-peer`. Generate a default config
there:
```
hbs2-peer init
```
This writes `~/.config/hbs2-peer/config` with sensible defaults: UDP
on 7351, TCP on 10351, key path `./default.key`. Open the file if you
want to see it; for this walkthrough no edits are needed.

`hbs2-peer init` does not generate the key file itself; the next step
does that.

## 2. Register the keyring directory

hbs2 ships with a small key manager (`hbs2-keyman`) that knows where
your local private keys live. Tools that need to sign something (the
peer, `git hbs2`, sync, and so on) ask the key manager rather than
reading files directly.

By default keyman scans `~/.hbs2-keyman/keys/`. Anything written
there (including keys auto-generated by `git hbs2 init --new`, see
step 5) is picked up automatically. The mask below is only needed
because the peer's own keyring will live in the `hbs2-peer` config
directory, outside the default scan path. Point keyman at it:
```
hbs2-keyman add-mask "$HOME/.config/hbs2-peer/*.key"
```
You only need to do this once per machine.

## 3. Generate a peer identity

The peer needs a long-term keypair to identify itself on the network.
Create it at the path the config expects, then have the key manager
rescan:
```
hbs2-cli hbs2:keyring:new > ~/.config/hbs2-peer/default.key
hbs2-keyman update
```
This file contains both the private and public halves of the keypair.
Keep it private.

For multi-machine setups you will later add either `known-peer`
entries pinning specific peers, or a `bootstrap-dns` line pointing
at a domain whose TXT record lists peer addresses. The hardcoded
default is `bootstrap.hbs2.app`; the domain is reserved but the
bootstrap node behind it is not yet deployed, so for now pin
specific peers with `known-peer` until that node is up.

## 4. Start the peer

In terminal 1:
```
hbs2-peer run
```
The peer starts up, opens its listening sockets, and waits. Leave it
running.

## 5. Push a git repository

A hbs2 git repository is identified by a public signing key. You do
not have to generate that key yourself; `git hbs2 init --new` creates
one, registers it with the key manager, and wires up a git remote in
one step.

Pick an existing git repository or create a small one. In terminal 2:
```
mkdir myrepo && cd myrepo
git init
echo hello > README.md
git add README.md
git commit -m "Initial commit"
```
Initialise the repository for hbs2:
```
git hbs2 init --new
```
The output looks like:
```
added git remote indicate-exist hbs23://8ZUnKpYfxTbDPQH85WMh9xmG5zXTVUeevsHexFxHLQAS
```
The base58 hash after `hbs23://` is the new repository's public
identifier (referred to below as `<REPO_KEY>`). The remote name
(`indicate-exist` in the example) is auto-generated as a memorable
two-word slug; yours will differ. List remotes any time with:
```
git hbs2 remotes
```
Then push to the auto-created remote:
```
git push indicate-exist
```
(substituting your own remote name). The push tells your local
`hbs2-peer` to publish the blocks into its storage.

## 6. Clone it back

To verify, clone the same repository into a separate directory:
```
cd /tmp
git clone hbs23://<REPO_KEY> myrepo-clone
cd myrepo-clone
cat README.md
```
You should see `hello`. The URL scheme is `hbs23://` (the binary that
git dispatches to is `git-remote-hbs23`). The clone talks to your
running `hbs2-peer` over its local socket and reads the blocks back
out.

## Where the keys live

There are two key files in play after the walkthrough above:

- **Peer identity** (step 3) — `~/.config/hbs2-peer/default.key`.
  You chose this path explicitly because it is what `hbs2-peer init`
  expects to find. Keyman discovers it through the mask you added in
  step 2.

- **Repository signing key** (step 5, auto-generated) —
  `~/.hbs2-keyman/keys/<REPO_KEY>-lwwref.key`, where `<REPO_KEY>`
  is the base58 pubkey printed by `git hbs2 init --new`. This is
  keyman's default keyring directory; no mask configuration is
  needed for keys that land there.

To see everything keyman currently knows about:
```
hbs2-keyman list
```
The output lists both private (signing-capable) and public-only
entries with their public keys and origin file paths.

The default keyring directory can be overridden by setting
`default-key-path "/some/path"` in `~/.config/hbs2-keyman/config`.
Any additional locations are added via `hbs2-keyman add-mask`.

Both files contain unencrypted private key material. Treat them like
SSH private keys: back them up where you would back up
`~/.ssh/id_ed25519`, and do not commit them to a git repository.

## Where to go next

- **Replicate to a second machine.** Repeat steps 1-4 on another
  machine, connect the two peers via `known-peer` or `bootstrap-dns`,
  and clone the repository on the second machine; pushes then
  replicate automatically. See
  [`docs/multi-machine.md`](docs/multi-machine.md) for the full
  walkthrough, or [`docs/MIRROR_SETUP.md`](docs/MIRROR_SETUP.md) for a
  dedicated mirror node with no checkout of its own.

- **Encrypted repositories.** Use group keys to make a repository
  readable only by holders of specific keys
  (`git hbs2 init --new --encrypted <group-key-hash>`). See
  [`docs/encrypted-repos.md`](docs/encrypted-repos.md) for the full
  walkthrough.

- **Architecture.** See [`ARCHITECTURE.md`](ARCHITECTURE.md) for a
  tour of the components involved in what you just did.
